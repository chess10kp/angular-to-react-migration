# PLAN.md

## Overview

The implementation follows `REPORT.md`'s hybrid model: deterministic inventory and codemods for repeatable transformations; agents only for ambiguous semantic rewrites and repair; mechanical build/test/browser gates; and human approval for routing, state, rollout, and contract changes. Existing files under `plans/` provide useful evidence-gating concepts, but they target AngularJS 1.x and explicitly do not apply directly to this Angular 17 codebase. They must be adapted rather than executed verbatim.

**Product framing.** The deliverable is a *reusable, intelligent Angular→React migration harness*, not a one-off rewrite. `harness-core` (scheduling, state, evidence gates, retries, replay, learning) is the product; the framework-general **Angular 2+ adapter** is its first proving ground. The harness-core and Angular 2+ adapter (below) must be built and pass their exit criteria before target-specific migration tasks begin. Success is measured by both migration parity *and* harness capability (see harness benchmark suite).

> **Target note.** The real pilot target is a private application we do **not** yet have access to.
> Earlier drafts of this plan used **OneCX** as a stand-in only to build the framework-general
> compiler; OneCX is **not** the real target and its specifics (topics, slots, preloaders,
> scope-polyfill, Keycloak, ngx-translate, `@onecx/*` packages) have been **removed**. The real
> target's ground-truth profile is captured in
> [`migration/real-target-discovery.md`](./migration/real-target-discovery.md) and summarized
> under "Real target profile" below. Until we can inventory the actual codebase, target-specific
> tasks stay generic and unknowns are marked explicitly.

## Goals

- Deliver a React 19 app that preserves observable behavior and all required `webcore`/auth integration contracts.
- Prove an incremental Angular-hosts-React seam before attempting a React-owned cutover.
- Keep the app functional as a micro-frontend under `webcore` throughout migration; do not disturb the global auth gateway or the sibling `admin` MFE.
- Build a reusable, idempotent codemod pipeline from TypeScript AST, Angular template AST, and React TSX transforms.
- Convert deterministic patterns automatically and emit typed, reviewable residue for semantic cases rather than silently guessing.
- Preserve or improve the existing (Jasmine) unit test coverage, then add browser-level parity for startup, routing, auth handoff, state (NgRx→RTK), and feature-flag decisions.
- Keep migration waves small using the Angular CLI/webpack project/lazy-route boundaries (the real target has **no Nx**; do not assume affected-graph sequencing).
- Make rollout reversible through route/entry flags (LaunchDarkly), a parallel React deployment, and a tested Angular fallback.
- Finalize one coherent React stack that feels familiar to Angular developers and reuses the vendor's React UI components.
- Measure success by behavior parity, repeatability, review effort, and safe delivery—not by the percentage of code generated by an LLM.

### Definition of done

- React owns the app's viewport and its routing within the `webcore` mount, without dual history ownership.
- The app still mounts correctly as a micro-frontend under `webcore` (auth handoff, env config, base path intact).
- Startup, auth handoff, route, permission, language (Transloco→i18next), state (NgRx→RTK), and feature-flag (LaunchDarkly) contracts pass parity tests.
- Production assets work under the real deployment base path, with Module Federation manifests, Docker/Nginx, and source-map policies validated.
- Angular code, Zone.js, Angular-only providers, and obsolete bridges are removed only after static and runtime use reach zero.
- The permanent React CI lane passes strict TypeScript, lint, unit/component, browser, accessibility, bundle, and production-artifact checks.

## P0 — Harness core and Angular 2+ adapter (build before any target-specific task)

> This section is the reusable engine. It must run end-to-end before the target profile below. The AngularJS-era designs under `plans/` (`00-ARCHITECTURE.md`, `01-STATE-AND-ARTIFACTS.md`, `02-TOOL-CONTRACTS.md`, `03-AGENT-ROLES.md`, `schemas/`) are ported and de-AngularJS'd here — their evidence-gating, ledger, lease, retry-budget, and recipe-induction principles are retained; their AngularJS-specific assumptions (`$scope`, controllers/directives, `angularjsVersion`, AngularJS DI/routers) are replaced.

- [ ] **Define the harness module boundary.** Split the system into `harness-core` (scheduling, unit state machine, evidence gates, retries, replay, learning), `source-angular` (Angular 17 compiler/`@if`-`@for` template-block/type-based-DI/route/RxJS analysis, plus feature-flag, translation-key, and HttpClient-interceptor extraction passes), `target-react` (TSX/routing/testing/build adapters), `profile-target` (the target's specifics: `webcore` mount, Okta handoff, NgRx→RTK, LaunchDarkly, Transloco, custom webpack plugins), `model-gateway` (provider-neutral structured agent calls), `oracle` (scenario execution, trace normalization, semantic diff), `recipe-registry` (codemods, exemplars, counterexamples, promotion), and `operator-control` (inspect, approve, retry, waive, abort, replay). Audit each concept in `plans/` and tag it core / angular-adapter / target-profile. No target-specific decision may live in `harness-core`.

- [ ] **Author framework-neutral artifact schemas + Angular 2+ adapters.** Create versioned, framework-neutral schemas plus Angular adapters for: `RunRequest`, `Unit`, `InventoryGraph`, `MigrationPlan`, `DecisionRecord`, `Patch`, `BehaviorScenario`, `SemanticTrace`, `EvidenceBundle`, `Counterexample`, `Recipe`, `RunManifest`, `RunResult`. Replace the AngularJS-encoded schemas under `plans/schemas/`. The orchestrator owns these; nothing downstream is built without them.

- [ ] **Implement the orchestrator engine.** Unit state machine, evidence-gated transitions, leases and optimistic revisions, bounded retries, role-scoped permissions, deterministic context packs, append-only ledger, and mechanical gate re-execution. No agent may drive a state transition; the orchestrator verifies artifacts and never trusts an agent's claim of success. Port from `plans/01`/`03`.

- [ ] **Specify the executable intelligence loop and its policies.** Observe → slice units and identify motifs → select deterministic recipe / agent task / abstention → produce a bounded plan → apply in an isolated workspace → validate mechanically → diagnose first meaningful divergence → repair within budget or escalate → convert successful repairs into candidate recipes → validate candidates on held-out fixtures before promotion. Write explicit policies for codemod-vs-agent selection, unit decomposition, tool ordering, confidence/evidence thresholds, abstention, human approval, and retry/escalation. Confidence in a transformation is distinct from a unit's risk score.

- [ ] **Define the behavior-oracle contract.** Framework-neutral semantic traces, trace normalization, settle-point semantics, configurable diff policies, counterexample bisection to the first divergent event, multi-run flake screening, fixture identity between Angular and React runs, mutation-kill thresholds, and waiver-aware equivalence. A failed parity check must yield a precise repair input, not merely "test failed."

- [ ] **Define the learning and recipe-promotion lifecycle.** counterexample → minimal diagnosis → repair → candidate lesson/recipe → generated regression fixture → replay against previous examples → held-out validation → human approval → versioned promotion. Define recipe invalidation when Angular/compiler versions change, the target stack changes, a counterexample contradicts a recipe, or model/tool versions alter success rates.

- [ ] **Guarantee replayability and provenance (P0, not deferred).** Every run emits a replay manifest: source commit + dirty-state hash, lockfile/config hashes, tool/compiler versions, model/provider/version + inference settings, prompt and context-pack hashes, recipe versions, sandbox image, environment/fixture identity, commands executed, patch and evidence hashes, and the exact replay command.

- [ ] **Enforce agent security and execution isolation (P0, not deferred).** Allowed write globs per role, forbidden source paths, command allowlists, network allowlists, secret/HAR redaction, CPU/time/token/output limits, transactional patch application, isolated worktrees or containers, no agent-controlled state transitions, and idempotency keys for mutating operations.

- [ ] **Define the operator interface.** `harness intake | inventory | plan [unit|wave] | run | inspect <run> | verify <run> | approve|reject|waive | retry|resume|abort | replay <manifest> | promote-recipe | dashboard`. Each review packet shows the plan, diff, evidence, confidence dimensions, unknowns, counterexamples, recipe provenance, and rollback action.

- [ ] **Build the harness benchmark suite (separate from migration parity).** Measure unit-decomposition correctness, recipe-selection accuracy, deterministic-transformation yield, agent-repair success, false-confidence rate, appropriate-abstention rate, repeat-run variance, cost/time per accepted unit, manual edit distance, post-promotion regression rate, and model-upgrade drift. Run against small synthetic motif fixtures **and at least one held-out Angular application distinct from the target**.

**Harness-core exit criteria (gate before the target profile):**

- The orchestrator state machine runs end-to-end.
- Angular 2+ schemas replace all AngularJS assumptions.
- Tool/model calls are typed, sandboxed, and replayable.
- One unit can move through observe → plan → convert → verify → repair/escalate.
- A counterexample can generate a candidate recipe and a regression fixture.
- The same core runs against the jHipster fixture **and** a second, distinct Angular conformance app.

## Migration focus (priority scope)

These are the concrete areas the harness must handle for this Angular 17 app; the codemod
and agent work is scoped to them first.

- **`@if` / `@for` control flow.** Target Angular 17's built-in control-flow blocks
  (`@if`/`@else`, `@for`, `@switch`) → JSX conditionals/`.map()`. Legacy `*ngIf`/`*ngFor`
  structural directives are **not** the focus; the source is already on the new syntax.
- **Translation service — rewrite, not port (Transloco).** The real target uses **Transloco**
  (`transloco` pipe, `*transloco` structural directive, `TranslocoService.translate/selectTranslate`,
  scoped/lazy translations) configured through `webcore` — **not** `@ngx-translate`. There is no
  drop-in equivalent; rewrite on the React side (i18next/react-i18next), preserving keys, scopes,
  interpolation params, and lazy loaders. The existing ngx-translate codemod slice does **not**
  match Transloco and needs a new adapter.
- **State — NgRx → Redux Toolkit.** The target uses NgRx plus direct mutations. Migrate each
  bounded store (actions/reducers/selectors/effects) to Redux Toolkit slices + RTK thunks/listeners,
  with parity tests over the same dispatched-action → state transitions.
- **Auth — Okta.** IdP is Okta; token refresh is interceptor-based. Use the Okta React SDK; do not
  assume Keycloak.
- **Service-closure for stateful components.** A component that injects Angular services cannot be
  lifted to React in isolation — its service dependency closure must be bridged (via the provider
  composition root / a typed adapter) or rewritten first. Treat each such component's service
  graph as part of its migration unit. (Whether the target uses modals/portals is not yet
  confirmed.)
- **Feature flags — LaunchDarkly.** The real target uses the **LaunchDarkly** SDK (not a bespoke
  3-mechanism subsystem). Migrate to the LaunchDarkly React SDK (`launchdarkly-react-client-sdk`),
  preserving flag keys, targeting context, and default values. Census every evaluation site (TS
  conditionals, route guards, `@if`/`*ngIf`-gated templates) so parity tests replay identical
  on/off decisions. (The bespoke plain/cohort/tenant-UUID model was an OneCX-profile assumption
  and does not apply here.)
- **HTTP → Axios** (see stack table): one shared Axios instance replacing `HttpClient` and its
  interceptors.
- **Retain: RxJS** (at `webcore`/remote integration boundaries, not as default component state —
  it's Redux Toolkit for store state), the **vendor UI library** (adopt its React components), and
  the **vendor/`metronome` design-system styles/tokens** during coexistence; migrate to
  component-local styles only after style-contract tests pass.
