# PLAN.md

## Overview

The implementation follows `REPORT.md`'s hybrid model: deterministic inventory and codemods for repeatable transformations; agents only for ambiguous semantic rewrites and repair; mechanical build/test/browser gates; and human approval for routing, state, rollout, and contract changes. Existing files under `plans/` provide useful evidence-gating concepts, but they target AngularJS 1.x and explicitly do not apply directly to this Angular 17 codebase. They must be adapted rather than executed verbatim.

**Product framing.** The deliverable is a *reusable, intelligent Angular→React migration harness*, not a one-off OneCX rewrite. `harness-core` (scheduling, state, evidence gates, retries, replay, learning) is the product; OneCX is **Pilot Profile 1** — its first adapter, proving ground, and conformance target. The harness-core and Angular 2+ adapter (below) must be built and pass their exit criteria before OneCX-specific migration tasks begin; the OneCX tasks then run as the first profile against that core. Success is measured by both migration parity *and* harness capability (see harness benchmark suite).

## Goals

- Deliver a React 19 shell that preserves observable behavior and all required OneCX integration contracts.
- Prove an incremental Angular-hosts-React seam before attempting a React-owned shell cutover.
- Keep existing Angular microfrontends functional during migration, including the Angular 18, 19, 20, and 21 preloaders.
- Build a reusable, idempotent codemod pipeline from TypeScript AST, Angular template AST, and React TSX transforms.
- Convert deterministic patterns automatically and emit typed, reviewable residue for semantic cases rather than silently guessing.
- Preserve or improve the existing 113 unit test cases, then add browser-level parity for startup, routing, topics, remotes, slots, and style isolation.
- Retain the current Nx control plane and use affected builds/tests to keep migration waves small.
- Make rollout reversible through route/entry flags, a parallel React shell deployment, and a tested Angular fallback.
- Finalize one coherent React stack that aligns with OneCX's published React packages and feels familiar to Angular developers.
- Measure success by behavior parity, repeatability, review effort, and safe delivery—not by the percentage of code generated by an LLM.

### Definition of done

- React is the sole owner of the shell viewport and browser history.
- Required Angular and non-Angular remotes still load through a documented compatibility matrix.
- Startup, auth, workspace, route, permission, theme, language, parameter, icon, slot, toast, and error contracts pass parity tests.
- Production assets work under the real deployment base path, with Module Federation manifests, Docker/Nginx, Helm, and source-map policies validated.
- Angular shell code, Zone.js, Angular-only providers, and obsolete bridges are removed only after static and runtime use reach zero.
- The permanent React CI lane passes strict TypeScript, lint, unit/component, browser, accessibility, bundle, and production-artifact checks.

### Final React stack decision

Use the stable OneCX React compatibility line available at implementation kickoff and pin the whole OneCX package family to one version. The currently compatible baseline is:

| Concern | Final choice | Angular-familiar rationale |
|---|---|---|
| Runtime | React 19.2.x, React DOM 19.2.x, strict TypeScript 5.9 | Current OneCX React 8.9 peer contract; typed function components and providers replace Angular components/providers. |
| Workspace/build | Nx 22.7.x + `@nx/react` + Vite 7 + `@module-federation/vite` 1.16 | Matches OneCX's current React generator direction while retaining Nx project graph and affected execution. Keep Webpack only for Angular preloaders during coexistence. |
| OneCX integration | `@onecx/react-integration-interface`, `@onecx/react-remote-components`, `@onecx/react-webcomponents`, `@onecx/react-utils`, `@onecx/react-auth`, plus framework-neutral `@onecx/accelerator` and `@onecx/integration-interface` | These are the supported React-side equivalents of the Angular-specific packages currently used. |
| Routing | React Router 7.13 data router | Route objects, lazy modules, loaders, redirects, and nested outlets map closely to Angular Router concepts. |
| UI/design | **Bootstrap 5** retained; **React-Bootstrap** for component behavior (modals, dropdowns, etc.), reusing the existing Bootstrap markup/classes and OneCX CSS variables | The source uses Bootstrap (via `ng-bootstrap`/`ngx-bootstrap`), so keep the same design system — React-Bootstrap maps the directive/component behavior to React with the same classes. Do not introduce PrimeReact or Tailwind. Verify exact React-Bootstrap/Bootstrap versions against the source. |
| Server state/API | **Axios** (shared instance with interceptors) + OpenAPI-generated typed client; an optional query/cache layer only where a feature needs it | Replaces `HttpClient` service injection, interceptors, retries, and request lifecycle. Axios is already a required peer of `@onecx/react-auth` (Keycloak), so it is the single HTTP client for the shell. |
| Platform/event state | OneCX React interfaces plus RxJS retained at topic/remote boundaries; thin typed hooks/adapters | Avoids rewriting cross-MFE protocols. RxJS remains infrastructure, not the default React component state model. |
| Client-only state | React state/reducer first; Zustand 5 only for genuinely shared shell-owned client state | Aligns with the OneCX React generator without introducing Redux for a source app that has no NgRx. If a later source contains NgRx, map that bounded store to Redux Toolkit instead. |
| DI/composition | Explicit TypeScript service interfaces and a small React provider composition root | Familiar provider boundary without adding Inversify or Effect-TS. Services remain testable and framework-independent. |
| Internationalization | i18next + react-i18next | `useTranslation()` and namespace loaders replace `TranslateService` and the translate pipe; the translation service is rewritten, preserving keys, namespaces, and lazy catalog loaders. |
| Auth | `@onecx/react-auth` + Keycloak 26 + its required Axios peer | Uses the platform-supported React auth layer rather than introducing a second OIDC abstraction. |
| Forms | React Hook Form + Zod, added only when a migrated feature actually has forms | Typed schema and controlled validation provide the closest practical equivalent to Angular reactive forms. This shell currently has no form migration need. |
| Testing | Vitest 4, React Testing Library 16, jest-dom, MSW, and Playwright | RTL replaces TestBed/harness tests; MSW stabilizes BFF contracts; Playwright supplies cross-framework behavior parity. |
| Styling | Retain the existing **Bootstrap compiled stylesheets / global CSS** during overlap; CSS Modules for new component-local styles | Keeps current Bootstrap classes and theme variables while preventing new global style coupling. Migrate off compiled CSS only after style-contract tests pass. |

`@onecx/react-generator` should be used as a reference or evaluated in a spike, not accepted blindly: its published generator line is newer/prerelease and may carry Nx/version assumptions that differ from the source. Do not adopt a prerelease OneCX 9 package family for production unless it is stable and compatibility-tested at kickoff.

## Tasks

## P0 — Harness core and Angular 2+ adapter (build before any OneCX task)

> This section is the reusable engine. It must run end-to-end before the OneCX profile below. The AngularJS-era designs under `plans/` (`00-ARCHITECTURE.md`, `01-STATE-AND-ARTIFACTS.md`, `02-TOOL-CONTRACTS.md`, `03-AGENT-ROLES.md`, `schemas/`) are ported and de-AngularJS'd here — their evidence-gating, ledger, lease, retry-budget, and recipe-induction principles are retained; their AngularJS-specific assumptions (`$scope`, controllers/directives, `angularjsVersion`, AngularJS DI/routers) are replaced.

- [ ] **Define the harness module boundary.** Split the system into `harness-core` (scheduling, unit state machine, evidence gates, retries, replay, learning), `source-angular` (Angular 17 compiler/`@if`-`@for` template-block/type-based-DI/route/RxJS analysis, plus feature-flag, modal service-closure, translation-key, and HttpClient-interceptor extraction passes), `target-react` (TSX/routing/testing/build adapters), `profile-onecx` (topics, preloaders, slots, style isolation, remotes), `model-gateway` (provider-neutral structured agent calls), `oracle` (scenario execution, trace normalization, semantic diff), `recipe-registry` (codemods, exemplars, counterexamples, promotion), and `operator-control` (inspect, approve, retry, waive, abort, replay). Audit each concept in `plans/` and tag it core / angular-adapter / onecx-profile. No OneCX-specific decision may live in `harness-core`.

- [ ] **Author framework-neutral artifact schemas + Angular 2+ adapters.** Create versioned, framework-neutral schemas plus Angular adapters for: `RunRequest`, `Unit`, `InventoryGraph`, `MigrationPlan`, `DecisionRecord`, `Patch`, `BehaviorScenario`, `SemanticTrace`, `EvidenceBundle`, `Counterexample`, `Recipe`, `RunManifest`, `RunResult`. Replace the AngularJS-encoded schemas under `plans/schemas/`. The orchestrator owns these; nothing downstream is built without them.

- [ ] **Implement the orchestrator engine.** Unit state machine, evidence-gated transitions, leases and optimistic revisions, bounded retries, role-scoped permissions, deterministic context packs, append-only ledger, and mechanical gate re-execution. No agent may drive a state transition; the orchestrator verifies artifacts and never trusts an agent's claim of success. Port from `plans/01`/`03`.

- [ ] **Specify the executable intelligence loop and its policies.** Observe → slice units and identify motifs → select deterministic recipe / agent task / abstention → produce a bounded plan → apply in an isolated workspace → validate mechanically → diagnose first meaningful divergence → repair within budget or escalate → convert successful repairs into candidate recipes → validate candidates on held-out fixtures before promotion. Write explicit policies for codemod-vs-agent selection, unit decomposition, tool ordering, confidence/evidence thresholds, abstention, human approval, and retry/escalation. Confidence in a transformation is distinct from a unit's risk score.

- [ ] **Define the behavior-oracle contract.** Framework-neutral semantic traces, trace normalization, settle-point semantics, configurable diff policies, counterexample bisection to the first divergent event, multi-run flake screening, fixture identity between Angular and React runs, mutation-kill thresholds, and waiver-aware equivalence. A failed parity check must yield a precise repair input, not merely "test failed."

- [ ] **Define the learning and recipe-promotion lifecycle.** counterexample → minimal diagnosis → repair → candidate lesson/recipe → generated regression fixture → replay against previous examples → held-out validation → human approval → versioned promotion. Define recipe invalidation when Angular/compiler versions change, the target stack changes, a counterexample contradicts a recipe, or model/tool versions alter success rates.

- [ ] **Guarantee replayability and provenance (P0, not deferred).** Every run emits a replay manifest: source commit + dirty-state hash, lockfile/config hashes, tool/compiler versions, model/provider/version + inference settings, prompt and context-pack hashes, recipe versions, sandbox image, environment/fixture identity, commands executed, patch and evidence hashes, and the exact replay command.

- [ ] **Enforce agent security and execution isolation (P0, not deferred).** Allowed write globs per role, forbidden source paths, command allowlists, network allowlists, secret/HAR redaction, CPU/time/token/output limits, transactional patch application, isolated worktrees or containers, no agent-controlled state transitions, and idempotency keys for mutating operations.

- [ ] **Define the operator interface.** `harness intake | inventory | plan [unit|wave] | run | inspect <run> | verify <run> | approve|reject|waive | retry|resume|abort | replay <manifest> | promote-recipe | dashboard`. Each review packet shows the plan, diff, evidence, confidence dimensions, unknowns, counterexamples, recipe provenance, and rollback action.

- [ ] **Build the harness benchmark suite (separate from migration parity).** Measure unit-decomposition correctness, recipe-selection accuracy, deterministic-transformation yield, agent-repair success, false-confidence rate, appropriate-abstention rate, repeat-run variance, cost/time per accepted unit, manual edit distance, post-promotion regression rate, and model-upgrade drift. Run against small synthetic motif fixtures **and at least one held-out, non-OneCX Angular application**.

**Harness-core exit criteria (gate before Pilot Profile 1):**

- The orchestrator state machine runs end-to-end.
- Angular 2+ schemas replace all AngularJS assumptions.
- Tool/model calls are typed, sandboxed, and replayable.
- One unit can move through observe → plan → convert → verify → repair/escalate.
- A counterexample can generate a candidate recipe and a regression fixture.
- The same core runs against OneCX **and** a small non-OneCX conformance app.

## Migration focus (priority scope)

These are the concrete areas the harness must handle for this Angular 17 shell; the codemod
and agent work is scoped to them first.

- **`@if` / `@for` control flow.** Target Angular 17's built-in control-flow blocks
  (`@if`/`@else`, `@for`, `@switch`) → JSX conditionals/`.map()`. Legacy `*ngIf`/`*ngFor`
  structural directives are **not** the focus; the source is already on the new syntax.
- **Translation service — rewrite, not port.** The `@ngx-translate` `TranslateService`/translate
  pipe has no drop-in equivalent; it is rewritten on the React side (i18next/react-i18next or an
  equivalent), preserving keys, namespaces, and lazy loaders. Runtime-loaded catalogs and
  interpolation params must survive.
- **Modals hold references to Angular services.** Modal components inject Angular services, so a
  modal cannot be lifted to React in isolation — its service dependencies must be bridged
  (exposed through the provider composition root / a typed adapter) or rewritten first. Treat
  each modal's service graph as part of its migration unit.
- **Feature-flag subsystem** must be migrated as a first-class concern, covering all three
  evaluation mechanisms:
  - simple enable/disable of a feature,
  - per-user **cohort** membership that turns features on for specific users,
  - **rules keyed on tenant UUID.**
  Provide a single React-side flag evaluator (hook + provider) that reproduces all three;
  parity tests must cover the same on/off decisions the Angular app makes.
- **HTTP → Axios** (see stack table): one shared Axios instance replacing `HttpClient` and its
  interceptors.
- **Retain: RxJS** (at OneCX topic/remote boundaries, not as default component state),
  **Bootstrap** (existing markup/classes preserved), and **compiled stylesheets** (keep the
  existing compiled CSS / theme output during coexistence; migrate to component-local styles
  only after style-contract tests pass).

## Notes/Assumptions

- `references/jhipster-ng17-fixture/` is the workflow-development source snapshot and green parity baseline (the real target codebase is private and not checked out); `references/onecx-shell-ui/` is retained only as a secondary Angular-21 reference. Implementation location/branching must be agreed before coding.
- This is Angular 17, not AngularJS. The existing `plans/` documents are not directly executable against it.
- The final shell may still need to host Angular microfrontends. “React migrated” means the shell itself no longer requires Angular; it does not automatically mean the wider OneCX ecosystem has no Angular remotes.
- Custom elements are the preferred early coexistence seam because the source already has a web-component route path and OneCX publishes React web-component utilities. A full single-spa layer is not planned unless the seam spike proves custom elements insufficient.
- Angular owns browser history until a dedicated, gated shell flip. Two global routers must never compete for `pushState`/`replaceState`.
- Use stable OneCX 8.x React packages as one pinned compatibility family unless a stable 9.x line is available and verified at kickoff. Avoid mixing stable and RC OneCX packages.
- Vite is the final React build target because it matches the current OneCX React generator and React-side Module Federation direction. Existing Angular preloaders can remain Webpack-built during coexistence; migrating them is not a prerequisite for the React scaffold.
- No Redux Toolkit or Effect-TS is planned for this shell because the current source has no NgRx and OneCX already exposes framework-neutral topics plus React adapters. Introduce Redux Toolkit only for a bounded NgRx migration discovered outside this snapshot.
- RxJS is retained for OneCX topic and remote protocols but should not become the default state solution inside new React presentation code.
- OpenAPI YAML remains the API source of truth. Generated Angular clients are replaced by regeneration, not line-by-line codemods.
- UI parity means user-visible behavior, accessibility, layout regions, design tokens, and integration events—not identical Angular DOM structure.
- Backend/auth access may be unavailable locally; deterministic MSW fixtures and production-like smoke environments are required to prevent this from blocking repeatable tests.
- Every loop is budgeted. Repeated identical failures, unknown public contracts, unsupported remote generations, or proposed product-visible behavior changes require escalation rather than autonomous retries.
- Human approval is mandatory for the charter, stack/version lock, remote compatibility strategy, router ownership cutover, intentional parity waivers, canary expansion, and Angular bridge deletion.
